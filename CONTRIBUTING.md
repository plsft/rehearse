# Contributing to GitGate

Thanks for taking the time to contribute. This guide covers the
**four open-source packages** in this repo, all Apache 2.0:

- [`@gitgate/runner`](runner) — local-first GitHub Actions runner (binary: `runner`)
- [`@gitgate/ci`](ts-ci) — TypeScript pipeline SDK that compiles to GitHub Actions YAML
- [`@gitgate/git-core`](git-engine) — pure-TypeScript git protocol implementation
- [`@gitgate/cli`](cli) — `gg`, the companion CLI for `@gitgate/ci`

There is no closed-source product. Everything in this repo ships under
Apache 2.0 to npm; the marketing site (`site/`) is private but contains
nothing proprietary.

## Development setup

```bash
# Requires Node 22+ and pnpm 9+
pnpm install
pnpm turbo build
pnpm turbo typecheck
pnpm turbo test
```

To work on a single package:

```bash
pnpm --filter @gitgate/runner test
pnpm --filter @gitgate/ci test
pnpm --filter @gitgate/git-core test
pnpm --filter @gitgate/cli dev -- ci compile   # run the gg CLI from source
```

To run the runner against a workflow during development:

```bash
node runner/dist/cli.js run .github/workflows/ci.yml
```

To run the cross-OS bench against `act`:

```bash
pnpm tsx bench/compare.ts --skip-cold
```

## Project layout

```
runner/        @gitgate/runner — the runner CLI + core
cli/           @gitgate/cli — the gg CLI (companion to @gitgate/ci)
ts-ci/         @gitgate/ci — TypeScript → GitHub Actions YAML SDK
git-engine/    @gitgate/git-core — pure-TS git engine

site/          marketing site at gitgate.com (private, deploys as Workers Static Assets)
bench/         cross-OS benchmark harness (runner vs act vs GitHub-hosted)
poc/           bench fixtures + a hono clone in poc/playground (gitignored)
scripts/       release tooling (release.mjs)
.gitgate/      our own CI pipelines authored in TypeScript
.github/       workflows generated by `gg ci compile` from .gitgate/pipelines/
```

## What we look for in a PR

1. **Tests.** Every public-API change has a matching test.
   - `@gitgate/runner` has 96 tests across 10 suites (matrix, expression,
     scheduler, cache, composite, artifacts, compat, js-action, reusable,
     setup-node).
   - `@gitgate/ci` has compiler snapshot tests under
     `ts-ci/test/compiler/snapshots/`. If your change updates YAML output,
     delete the affected snapshot and re-run `pnpm --filter @gitgate/ci
     test` to regenerate; commit both.
   - `@gitgate/git-core` has 162 tests across 8 suites.
2. **Typecheck clean.** `pnpm turbo typecheck` must pass.
3. **No new runtime deps in `@gitgate/ci`.** The compiled YAML must work
   without GitGate at runtime — that means the SDK itself can't drag in
   anything beyond what's in the runtime contract. Dev-only deps are fine.
4. **No native deps in `@gitgate/git-core`.** It runs on Cloudflare
   Workers, the browser, Node, and Bun. `pako` is the only allowed
   runtime dep.
5. **No regressions in the runner's compat or bench numbers.** If you're
   touching the runner core, run
   `node runner/dist/cli.js compat poc/playground/hono/.github/workflows/ci.yml`
   and `pnpm tsx bench/compare.ts --skip-cold` — both should be roughly
   stable. `bench/RESULTS.md` is the source of truth for the marketing
   site's claims.
6. **Conventional commits.** `feat(scope):`, `fix(scope):`, `test(scope):`,
   `docs:`, `chore:`, `bench:`, `release:`. The `scope` is the package or
   area being touched (`runner`, `ci`, `git-core`, `cli`, `site`, `bench`).

## Releasing

Releases are gated by the `scripts/release.mjs` helper, which keeps all
four package.json versions in lockstep. From a clean working tree:

```bash
pnpm release patch        # 0.3.2 → 0.3.3
pnpm release minor        # 0.3.x → 0.4.0
pnpm release major        # 0.x → 1.0.0
pnpm release prerelease   # → 0.3.3-next.0
```

The script bumps versions, commits as `release: vX.Y.Z`, tags `vX.Y.Z`,
and pushes. The release.yml workflow on GitHub takes the tag from there
and publishes all four packages with the right npm dist-tag (`latest`
for stable, `next`/`rc`/`beta` for prereleases).

## Reporting issues

- **Bug reports** — minimal repro plus your Node / pnpm / OS versions and
  the workflow YAML you ran.
- **Compat regressions** — if `runner compat <yourworkflow>` reports a
  step as `uses-unsupported` and you think it should be supported, please
  open an issue with the action ref and version.
- **Bench regressions** — if a bench number drifts noticeably from
  `bench/RESULTS.md`, mention the OS, the warm-cache state, and how you
  reproduced.
- **Security** — please email <george@flarefound.com> rather than opening
  a public issue.
