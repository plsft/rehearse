# Contributing to Rehearse

Thanks for taking the time to contribute. This guide covers the
**four open-source packages** in this repo, all Apache 2.0:

- [`@rehearse/cli`](runner) ŌĆö local-first GitHub Actions runner (binary: `rh`)
- [`@rehearse/ci`](ts-ci) ŌĆö TypeScript pipeline SDK that compiles to GitHub Actions YAML
- [`@rehearse/git-core`](git-engine) ŌĆö pure-TypeScript git protocol implementation
- [`@rehearse/cli`](cli) ŌĆö `rh`, the companion CLI for `@rehearse/ci`

There is no closed-source product. Everything in this repo ships under
Apache 2.0 to npm; the marketing site (`site/`) is private but contains
nothing proprietary.

## Development setup

```bash
# Requires Node 22+ and pnpm 9+
pnpm install
pnpm turbo build
pnpm turbo typecheck
pnpm turbo test
```

To work on a single package:

```bash
pnpm --filter @rehearse/cli test
pnpm --filter @rehearse/ci test
pnpm --filter @rehearse/git-core test
pnpm --filter @rehearse/cli dev -- ci compile   # run the rh CLI from source
```

To run the runner against a workflow during development:

```bash
node runner/dist/cli.js run .github/workflows/ci.yml
```

To run the cross-OS bench against `act`:

```bash
pnpm tsx bench/compare.ts --skip-cold
```

## What we look for in a PR

1. **Tests.** Every public-API change has a matching test.
   - `@rehearse/cli` has 121 tests across 13 suites (matrix, expression,
     scheduler, cache, composite, artifacts, compat, js-action, reusable,
     setup-node, setup-dotnet, image-resolver, git-context).
   - `@rehearse/ci` has compiler snapshot tests under
     `ts-ci/test/compiler/snapshots/`. If your change updates YAML output,
     delete the affected snapshot and re-run `pnpm --filter @rehearse/ci
     test` to regenerate; commit both.
   - `@rehearse/git-core` has 162 tests across 8 suites.
2. **Typecheck clean.** `pnpm turbo typecheck` must pass.
3. **No new runtime deps in `@rehearse/ci`.** The compiled YAML must work
   without Rehearse at runtime ŌĆö that means the SDK itself can't drag in
   anything beyond what's in the runtime contract. Dev-only deps are fine.
4. **No native deps in `@rehearse/git-core`.** It runs on Cloudflare
   Workers, the browser, Node, and Bun. `pako` is the only allowed
   runtime dep.
5. **No regressions in the runner's compat or bench numbers.** If you're
   touching the runner core, run
   `node runner/dist/cli.js compat poc/playground/hono/.github/workflows/ci.yml`
   and `pnpm tsx bench/compare.ts --skip-cold` ŌĆö both should be roughly
   stable. `bench/RESULTS.md` is the source of truth for the marketing
   site's claims.
6. **Conventional commits.** `feat(scope):`, `fix(scope):`, `test(scope):`,
   `docs:`, `chore:`, `bench:`, `release:`. The `scope` is the package or
   area being touched (`runner`, `ci`, `git-core`, `cli`, `site`, `bench`).

## Releasing

Releases are gated by the `scripts/release.mjs` helper, which keeps all
four package.json versions in lockstep. From a clean working tree:

```bash
pnpm release patch        # 0.3.2 ŌåÆ 0.3.3
pnpm release minor        # 0.3.x ŌåÆ 0.4.0
pnpm release major        # 0.x ŌåÆ 1.0.0
pnpm release prerelease   # ŌåÆ 0.3.3-next.0
```

The script bumps versions, commits as `release: vX.Y.Z`, tags `vX.Y.Z`,
and pushes. The release.yml workflow on GitHub takes the tag from there
and publishes all four packages with the right npm dist-tag (`latest`
for stable, `next`/`rc`/`beta` for prereleases).

## Reporting issues

- **Bug reports** ŌĆö minimal repro plus your Node / pnpm / OS versions and
  the workflow YAML you ran.
- **Compat regressions** ŌĆö if `rh compat <yourworkflow>` reports a
  step as `uses-unsupported` and you think it should be supported, please
  open an issue with the action ref and version.
- **Bench regressions** ŌĆö if a bench number drifts noticeably from
  `bench/RESULTS.md`, mention the OS, the warm-cache state, and how you
  reproduced.
- **Security** ŌĆö please email <george@flarefound.com> rather than opening
  a public issue.
