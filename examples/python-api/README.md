# python-api — FastAPI + pytest + Postgres service

A minimal FastAPI app backed by Postgres, with integration tests that hit a
real database. The CI pipeline starts Postgres as a service container,
then runs `pytest`.

## What this example demonstrates

| Feature | Where |
| --- | --- |
| `services:` block | Postgres 16-alpine starts as a sidecar |
| Container backend (auto-selected when `services:` present) | `runner` picks `container` automatically |
| `--network-alias` networking | tests reach DB at `postgres:5432` from inside the job container |
| Health-check waiting | runner waits for `pg_isready -U postgres` before first step |
| `actions/cache` for pip | `~/.cache/pip` keyed on `requirements.txt` hash |
| Python preset | `python.setup('3.12')` → `actions/setup-python@v5` |

This is the workflow class **`act` doesn't complete** — it times out at 360s
on Linux and Windows GH-runners. `@rehearse/cli` finishes it in ~12s.

## Pipeline at a glance

```typescript
job('test', {
  runner: Runner.github('ubuntu-latest'),
  services: {
    postgres: {
      image: 'postgres:16-alpine',
      env: { POSTGRES_USER: 'postgres', POSTGRES_PASSWORD: 'postgres', POSTGRES_DB: 'postgres' },
      ports: ['5432:5432'],
      options: '--health-cmd "pg_isready -U postgres" --health-interval 5s --health-timeout 5s --health-retries 10',
    },
  },
  env: {
    DATABASE_URL: 'postgresql+psycopg://postgres:postgres@postgres:5432/postgres',
  },
  steps: [
    step.checkout(),
    python.setup('3.12'),
    step.cache({ path: '~/.cache/pip', key: `pip-\${{ runner.os }}-${hashFiles('**/requirements.txt')}`, restoreKeys: ['pip-${{ runner.os }}-'] }),
    step.run('pip install -r requirements.txt', { name: 'Install dependencies' }),
    step.run('pytest', { name: 'Run tests' }),
  ],
}),
```

## Run locally

```bash
cd examples/python-api

# Make sure Docker Desktop is running (container backend needs it)
docker version

# Install the Rehearse toolchain
npm install -g @rehearse/cli
npm install                  # gets @rehearse/ci as a devDep

# Compile + run
rh ci compile
rh run .github/workflows/ci.yml --backend container
```

Expected: ~10–15s warm. The runner pulls `postgres:16-alpine`, starts it
on a private Docker network, waits for the health check to go green,
then runs `pytest` in a sibling container that resolves `postgres` as a
hostname via `--network-alias`.

## Compare with `act`

Try the same workflow with `act`:

```bash
act -W .github/workflows/ci.yml -j test
```

`act` will start `postgres:16-alpine`, log it as healthy, then hang
indefinitely on `pg_isready -h postgres` from the job container — its
service-network plumbing has a real bug. Bench documented at
[`bench/RESULTS.md`](../../bench/RESULTS.md).
