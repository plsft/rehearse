How the scoreboard updates.

Three triggers, all driving the same bench/compat/run.mjs. The result JSON is committed back to main after every run, which is what this page reads.

Nightly · 07:00 UTC

A fresh GH-hosted ubuntu runner installs Node + bun, runs npx -y @rehearse/cli@latest run … against every fixture, scores each against its declared expectation, commits the JSON.

PR gate

Every PR that touches cli/, ts-ci/, or bench/compat/ runs the same scoreboard. A PR that drops any fixture's expected outcome fails CI and can't merge until either (a) the regression is fixed, or (b) the fixture is honestly updated to match new reality.

Manual
gh workflow run compat.yml \
  --ref main \
  -f cli=@rehearse/cli@0.6.18

Honest framing.

  • — Four fixtures is small. This is a starting set focused on the surfaces where compat regressions historically hurt: matrix execution (typey), source-tree walking under matrix worktrees (kleur), workflow_dispatch input substitution (synthetic), and the host-shim happy path (is-plain-obj). Adding a fixture is a deliberate act — we add when reality changes, never to soften an assertion.
  • — "Pass" doesn't mean "100% bit-equivalent." Each fixture declares one of: pass (workflow exits 0), no-eloop (no infinite recursion in the log), input-substituted (workflow_dispatch input round-trips). The runner's source is in the repo; nothing is hidden.
  • — Failing the gate is the point. The first nightly run flagged a real customer-facing bug (host-tool shims silently lying when bun/python/etc. wasn't on PATH). Fixed in v0.6.18 with regression tests. That's the value: catching compat drift the moment it lands, not when a customer files an issue.
