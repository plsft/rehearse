Release notes.

All four packages ship in lockstep at the same version. Tags live at github.com/plsft/rehearse/releases.

Pro daemon v0.2.2

2026-05-06 · server-side fix · no client upgrade needed
Fixed: silent stall on --remote runs

A subset of rh run --remote requests stalled with no output past the initial [remote] clone… line. Root cause: the sprite VM image uses nvm, which installs npm-globals to a path the daemon's systemd PATH excluded — so the daemon's bash -c "rh run …" couldn't resolve rh, and on this image the missing-binary case manifested as a silent shell hang rather than a clean exit-127.

Fix
  • — Daemon resolves rh's absolute path at startup via npm config get prefix; uses it directly when spawning instead of relying on PATH.
  • — Prepends npm-global-bin to the workflow runner env so any other npm-installed tool a workflow expects is also reachable.
  • — ensureCliInstalled now logs npm errors to the daemon journal; previously failures were silent.
  • — Hard-fails with a clear stream message + exit 127 if rh is missing after install — no more silent voids.
Validated

3 back-to-back --remote runs against typey ci.yml: each completed in vm=3.1-3.2s wall=4.5-4.9s with full logs persisted. Cold-wake budget (60s, from v0.6.15) confirmed adequate; no transient 500s observed.

Customer impact

No CLI upgrade required — daemon is server-side. Rolled to all customer sprites manually for now; new provisions pick it up automatically.

v0.6.17

2026-05-06 · Round 3 OSS validation fix — worktree ELOOP
Fixed: source-tree walk ELOOP

Workflows whose steps walk the source tree (coverage tools like c8, linters, file-globbing scripts) hit Windows ELOOP after ~70 levels because the per-cell worktree lived at <repo>/.runner/worktrees/<id>/ and the worktree symlinked .runner/ back to the parent repo. Fixed by removing .runner/ from the worktree symlink list. Per-cell caches (npm/yarn/pip) are unaffected — they use absolute env-var paths set by the host backend.

Surfaced by

Round 3 validation against lukeed/kleur's CI. The kleur workflow runs c8 npm test for coverage; c8 walked into .runner/worktrees/test/.runner/worktrees/test/... indefinitely on Windows. Reproduced 100%, fixed in v0.6.17.

v0.6.16

2026-05-06 · agent-ci gap closure — features for real-world workflows
Remote reusable workflows

Workflows that call uses: org/repo/.github/workflows/x.yml@ref at the job level now run instead of skip-with-reason. Target repo is shallow-cloned into the shared action cache; nested ./ refs inside the called workflow resolve relative to the cloned repo. Caller's with: and secrets: still substitute correctly.

Reusable workflow outputs flow

A reusable workflow's on.workflow_call.outputs: declarations now propagate to caller dependents. ${{ needs.<callerKey>.outputs.<X> }} resolves against the inner job results via a synthetic "umbrella" job the planner emits per caller. Inner failure propagates to the umbrella so downstream needs: short-circuits properly.

Shared action cache

Action repos (actions/checkout, setup-node, etc.) now cache to ~/.rehearse/actions-cache/ instead of .runner/actions/ per repo. Override with REHEARSE_ACTIONS_CACHE. Second run of the same action from any repo on the same host is a no-op resolve. Falls back to per-repo if HOME is unwritable (CI sandboxes).

workflow_dispatch input prompts
# before: missing required inputs collapsed to '' silently
rh run .github/workflows/release.yml

# now: prompted in TTY, validated, fail-loud in CI
rh run .github/workflows/release.yml --input env=staging --input tag=v1.2.3

Resolution order: CLI flag → API option → action.yml default → interactive prompt → error. Type validation for boolean, number, and choice. Non-TTY environments get a clear "missing required input" error instead of hanging on stdin.

v0.6.15

2026-05-06 · stability — recursive reusables, --collapse-matrix, sprite cold-wake
Recursive reusable workflows + cycle detection

Reusable expansion now recurses up to the GitHub Actions spec limit of 4 levels. A self-referential workflow throws cycle detected in reusable workflows instead of hanging the planner. Composite job keys accumulate (caller__inner__deeper__terminal).

--collapse-matrix

Collapses an N-cell matrix to a single representative cell (first value of each variable). For fast local iteration on multi-cell workflows when you just want a green/red signal once. Composes with --matrix.

Pro — sprite cold-wake budget 12s → 60s

Round 2 validation hit transient 500s on back-to-back remote runs when the sprite went idle between runs. API now retries the wake-up handshake for up to 60s (was 12s) before surfacing the error. Eliminated the cold-wake regression seen in three of six sequential --remote runs.

We now set BOTH the canonical (INPUT_INHERIT-TOOLCHAIN) and legacy underscore variant per dashed input.

v0.6.0

2026-05-06 · breaking — single binary rh
⚠ Breaking — package structure
  • — Two npm packages now, not four: @rehearse/cli (the binary rh) and @rehearse/ci (the TypeScript SDK). @rehearse/runner and @rehearse/git-core are no longer published — folded into @rehearse/cli.
  • — One binary, not two: rh. Old rehearse binary is gone. rh run, rh watch, rh compat, rh install-hook for the runner; rh ci compile, rh ci convert, rh ci init, rh ci validate, rh ci watch, rh ci estimate for TS-pipeline ergonomics.
Migration
# before
npm install -g @rehearse/runner @rehearse/cli
rehearse run .github/workflows/ci.yml
rh ci compile

# after
npm install -g @rehearse/cli
rh run .github/workflows/ci.yml
rh ci compile
Why

The two-package, two-binary split was a historical accident from the project's pivot. @rehearse/cli contained the companion CLI (rh) while @rehearse/runner contained the main binary (rehearse) — backwards from intuition. Now @rehearse/cli is THE Rehearse CLI, and @rehearse/ci is the SDK you import when you author pipelines in TypeScript.

Pre-push hook

Existing .git/hooks/pre-push from v0.5.x still references @rehearse/runner and will continue to work via the v0.5.4 npm package (frozen forever). Re-run rh install-hook after upgrading to switch to the new npx -p @rehearse/cli rh run form.

v0.5.4

2026-05-06 · matrix install correctness
Runner — matrix scratch caches
  • — Matrix cells now write to isolated per-cell package-manager caches for race-prone managers (npm, yarn, pip). Path: .runner/cache/<cellId>/{npm,yarn,pip}/ — deterministic so caches warm across runs of the same matrix configuration.
  • — Content-addressed managers (pnpm, bun, cargo, Go) keep using the user's global cache — they're parallel-safe by design and benefit from existing warm state.
  • — Each matrix cell now installs into its own worktree's node_modules (previously symlinked to the parent, which let parallel npm install cells trample each other's tree on the shared path).
  • — git worktree add now robust against stale registrations (fires git worktree remove --force + prune before adding, so rm -rf .runner/ between runs no longer corrupts state).
Validated on fastify (Windows x64)
  • — 2-cell test-pino-compat matrix: PASS in 1m44s, zero TAR-rename errors.
  • — Cold install per cell: ~40.7s. Warm install per cell (second run): ~31.6s — per-cell cache reuse confirmed.
  • — Eliminates the npm warn tar TAR_ENTRY_ERROR floods previously seen on parallel npm install on Windows.

v0.5.3

2026-05-06 · reusable workflows + matrix backend
Three bugfixes from fastify CI audit
  • — rehearse compat no longer crashes with "Cannot read properties of undefined (reading 'map')" when a workflow has reusable-workflow callers (job-level uses:). Synthesizes a single reusable-local or reusable-remote step entry so the audit completes.
  • — Backend selection now reads the matrix-substituted runs-on. Previously a Windows host running a job with runs-on: ${{ matrix.os }} would always pick host backend regardless of cell target. Now correctly falls back to container for cross-OS cells.
  • — Remote reusable workflows (org/repo/.github/workflows/x.yml@ref) now show a clear ⊘ remote reusable workflow not supported: … skip line at run time, instead of phantom-running with 0 steps.
UX
  • — --job <key> filter now also matches expanded reusable inner jobs by caller-key prefix (e.g. --job coverage-nix matches coverage-nix__check-coverage).
