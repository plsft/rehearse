// isolated

Single-tenant VM. Not a shared pool.

One dedicated microVM per customer — not a shared host, not a container on someone else's kernel, not a sticky-disk slice. Hardware-level isolation. Your code never lands on another customer's box. Real security boundary for compliance-conscious teams.

// same cli, three targets

One binary, three execution targets.

rh run for local. rh run --remote for Pro. git push for GitHub Actions. Same YAML, same output format, same exit codes. Switch with one flag — no config files, no second tool, no lock-in. Drop the flag and you're back to free OSS execution.

// streaming logs

Live output, no buffering.

Daemon streams stdout and stderr line-by-line over a long-lived HTTP connection. Your terminal updates as the workflow runs, not at the end. Closes with the real exit code so your shell scripts work.

// secrets via --env-file

Ship secrets, deploy from CI.

--env-file .env ships your local credentials to the VM as ${{ secrets.* }}. Deploy to AWS / Cloudflare / Fly without provisioning anything in CI itself. The VM has curl / ssh / rsync; install aws / terraform on first use, cached forever after.

// roadmap

The four open gaps, with a plan for each.

Every item here is in active design or scoping. None of them block the OSS runner locally — they're specifically about what Pro's hosted VM can run today vs. what a GH-hosted runner can run.

①
SERVICES ON PRO
Postgres / Redis / MySQL service containers.

jobs..services: works locally on the OSS runner via the container backend, but Pro VMs don't expose Docker today. Need either Docker-in-VM or a per-team service-broker.

target: v1 · workaround: use OSS runner locally for service workflows
②
CONCURRENCY: CANCELLATION
concurrency: deploy-prod with cancel-in-progress.

The concurrency: block is parsed but not enforced — runs don't cancel each other when a new run lands in the same group. Required for safe deploy queues. Single-VM v0 makes this naturally serialized; the cancel-in-progress semantics need orchestrator work.

target: v1 · workaround: sequential dispatch, dedup before invoking
③
REMOTE REUSABLE WORKFLOWS
uses: org/repo/.github/workflows/foo.yml@ref

Local uses: ./.github/workflows/foo.yml works; the remote variant doesn't yet (would need git-fetch + parse the upstream workflow + recurse). Composite actions already do remote auto-clone, so the path is well-trodden — just hasn't been written for reusable workflows.

target: v1 · workaround: inline the called workflow into your repo
④
OIDC TOKEN ISSUER
permissions: id-token: write + role-to-assume.

aws-actions/configure-aws-credentials@v4 with role-to-assume needs a JWT from an OIDC issuer that AWS trusts. GitHub-hosted runs get this from token.actions.githubusercontent.com; Pro VMs don't have that. Plan: stand up oidc.rehearse.sh with a Pro-team-scoped issuer that customers add to AWS / Azure / GCP as a trusted IdP. One-time setup per cloud.

target: v1.1 · workaround: long-lived IAM keys via --env-file (works today)

Tracking these in github.com/plsft/rehearse/issues — comment if any of these blocks you and we'll prioritize accordingly.

// deploy anywhere

Deploy from your CI, not just build.

Pass long-lived credentials with --env-file; they ship to your VM and become ${{ secrets.* }} for workflow expansion. Install whatever cloud CLI you need on-demand inside the workflow (pip wheel, npm install). Verified end-to-end against AWS STS today.

# 1. Put your cloud creds in a local .env (NEVER commit it)
$ cat .env
AWS_ACCESS_KEY_ID=AKIA...
AWS_SECRET_ACCESS_KEY=...
AWS_DEFAULT_REGION=us-east-1

# 2. Same workflow you'd run on GitHub-hosted
$ rh run --remote --env-file .env .github/workflows/deploy.yml

Cloud CLIs: the VM ships curl, wget, rsync, ssh. Install aws/az/gcloud/kubectl/terraform in the workflow itself — pip-wheel installs (~5s) cache to your persistent rootfs after the first run.

Honest gap: OIDC-based auth (permissions: id-token: write + aws-actions/configure-aws-credentials with role-to-assume) doesn't work — there's no GitHub OIDC issuer in front of your VM. Use long-lived credentials for now. OIDC bridge is a v1 candidate.

// measured · not marketing

Real numbers on the same workload.

A workflow that runs npm install typescript vite vitest react react-dom express axios zod hono — eight popular packages, no project deps. Three scenarios, all measured this morning.

scenario install wall notes
Local Docker, cold pull + install — 22.0s first docker run on a clean machine
Local Docker, image cached 17.1s 17.1s image cached, no npm cache
Pro cold cache 10.5s 12.5s first run on a fresh VM
Pro warm cache 5.1s 6.7s second + subsequent runs

Pro warm vs local Docker cached: 2.55× faster. Cold-to-warm Pro speedup: 2.07× on install, 1.86× on wall. The real savings show up when you run CI dozens of times a day, which is the whole point of CI.

Reproduce: rh run --remote bench/pro-bench.yml against your Pro account, twice. The first run is cold; the second is warm. Exact ms vary by region and upstream-registry latency — the cold/warm ratio is what's load-bearing.

// honest caveats

What Pro is not.

  • — 11 languages today, PHP not yet. Pro v0 verified end-to-end against your Pro account on Node, Bun, Deno, Python, Go, Java, Ruby, Rust, Elixir, Erlang, and .NET. PHP is the only language not yet supported (no managed install path on the VM today); those workflows still run great on the OSS runner locally.
  • — No service containers (yet). Workflows that declare services: (Postgres, Redis, etc.) need Docker on the VM, which isn't available on the default tier. Use the OSS runner with the container backend locally for those.
  • — First run isn't faster. The cache populates on run 1; runs 2-N benefit. If your project sees its first CI run on Pro and never again, you'll see no win.
  • — Not a parallel-matrix multiplier. One VM per team in v0. Matrix jobs run sequentially on your VM. Team plans add parallelism in v1.
  • — Caches are private to your VM. No cross-team sharing — that's the security boundary. If you have 5 unrelated repos, all 5 share one cache, but no other customer touches yours.
  • — Not free. $49/mo. The OSS runner is and always will be Apache 2.0 and free.
