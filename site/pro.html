// roadmap

The four open gaps, with a plan for each.

Every item here is in active design or scoping. None of them block the OSS runner locally — they're specifically about what Pro's hosted sprite can run today vs. what a GH-hosted runner can run.

①
SERVICES ON PRO
Postgres / Redis / MySQL service containers.

jobs..services: works locally on the OSS runner via the container backend, but Pro sprites don't expose Docker today. Need either Docker-in-sprite or a per-team service-broker.

target: v1 · workaround: use OSS runner locally for service workflows
②
CONCURRENCY: CANCELLATION
concurrency: deploy-prod with cancel-in-progress.

The concurrency: block is parsed but not enforced — runs don't cancel each other when a new run lands in the same group. Required for safe deploy queues. Single-sprite v0 makes this naturally serialized; the cancel-in-progress semantics need orchestrator work.

target: v1 · workaround: sequential dispatch, dedup before invoking
③
REMOTE REUSABLE WORKFLOWS
uses: org/repo/.github/workflows/foo.yml@ref

Local uses: ./.github/workflows/foo.yml works; the remote variant doesn't yet (would need git-fetch + parse the upstream workflow + recurse). Composite actions already do remote auto-clone, so the path is well-trodden — just hasn't been written for reusable workflows.

target: v1 · workaround: inline the called workflow into your repo
④
OIDC TOKEN ISSUER
permissions: id-token: write + role-to-assume.

aws-actions/configure-aws-credentials@v4 with role-to-assume needs a JWT from an OIDC issuer that AWS trusts. GitHub-hosted runs get this from token.actions.githubusercontent.com; Pro sprites don't have that. Plan: stand up oidc.rehearse.sh with a Pro-team-scoped issuer that customers add to AWS / Azure / GCP as a trusted IdP. One-time setup per cloud.

target: v1.1 · workaround: long-lived IAM keys via --env-file (works today)

Tracking these in github.com/plsft/rehearse/issues — comment if any of these blocks you and we'll prioritize accordingly.

// deploy anywhere

Deploy from your CI, not just build.

Pass long-lived credentials with --env-file; they ship to your sprite and become ${{ secrets.* }} for workflow expansion. Install whatever cloud CLI you need on-demand inside the workflow (pip wheel, npm install). Verified end-to-end against AWS STS today.

# 1. Put your cloud creds in a local .env (NEVER commit it)
$ cat .env
AWS_ACCESS_KEY_ID=AKIA...
AWS_SECRET_ACCESS_KEY=...
AWS_DEFAULT_REGION=us-east-1

# 2. Same workflow you'd run on GitHub-hosted
$ runner run --remote --env-file .env .github/workflows/deploy.yml

Cloud CLIs: sprite ships curl, wget, rsync, ssh. Install aws/az/gcloud/kubectl/terraform in the workflow itself — pip-wheel installs (~5s) cache to your persistent rootfs after the first run.

Honest gap: OIDC-based auth (permissions: id-token: write + aws-actions/configure-aws-credentials with role-to-assume) doesn't work — there's no GitHub OIDC issuer in front of your sprite. Use long-lived credentials for now. OIDC bridge is a v1 candidate.

// honest caveats

What Pro is not.

  • — 11 languages today, PHP not yet. Pro v0 verified end-to-end against your Pro account on Node, Bun, Deno, Python, Go, Java, Ruby, Rust, Elixir, Erlang, and .NET. PHP is the only language not yet supported (Sprites.dev doesn't ship PHP and there's no clean install path); those workflows still run great on the OSS runner locally.
  • — No service containers (yet). Workflows that declare services: (Postgres, Redis, etc.) need Docker on the sprite, which isn't available on the default tier. Use the OSS runner with the container backend locally for those.
  • — First run isn't faster. The cache populates on run 1; runs 2-N benefit. If your project sees its first CI run on Pro and never again, you'll see no win.
  • — Not a parallel-matrix multiplier. One sprite per team in v0. Matrix jobs run sequentially on your sprite. Team plans add parallelism in v1.
  • — Caches are private to your sprite. No cross-team sharing — that's the security boundary. If you have 5 unrelated repos, all 5 share one cache, but no other customer touches yours.
  • — Not free. $49/mo. The OSS runner is and always will be Apache 2.0 and free.
