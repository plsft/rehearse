// Bench

Real workflows. Real numbers.

Two comparisons. Rehearse vs GitHub Actions (the comparison most users care about) and Rehearse vs act (the OSS-vs-OSS comparison). Both reproducible, methodology below.

// Headline · Same source, three execution targets

A real 9-cell matrix workflow.

plsft/typey — a real Bun CLI repo. CI runs 3 OS × 3 Bun versions = 9 cells, each with checkout / setup-bun / install / format / lint / typecheck / test / build. The same compiled YAML on three execution targets:

LOCAL · rehearse run
1.5s

Sub-second per cell, warm. Worker concurrency capped at 4.

PRO · rehearse run --remote
5s

Single-tenant VM. Network roundtrip + actual VM execution. Cells serialize on Pro to avoid bun-cache install race.

GH ACTIONS
95s

9 fresh containers, each downloads Bun cold, each cold-loads every tool. Wall = slowest cell.

63× faster local · 19× faster on Pro vs GitHub Actions on the same workflow. Numbers measured 2026-05-05 on v0.4.0 against plsft/typey@6595d95.

// Methodology

Reproducibility checklist.

Environment

  • GitHub-hosted ubuntu-latest runner (2-core, 7 GB RAM, free tier).
  • For the typey 9-cell matrix: 3 OSs (ubuntu-latest, macos-latest, windows-latest) × 3 Bun versions (1.2.x, 1.3.x, latest).
  • Local Rehearse run on a Windows 11 laptop, host backend, default --max-parallel 4.
  • Pro Rehearse run via rehearse run --remote against a default-tier microVM.

Methodology

  • Warm-cache numbers are the second run, with deps already on disk. Cold-vs-cold is narrower because a fresh pnpm install dominates everything.
  • Each comparison is the same YAML file with no Rehearse-specific edits — the bench compares execution paths, not config rewrites.
  • act timed against the same YAML with --no-cache-server + -P label mappings.
  • Pro VM cells serialize via --max-parallel 1 on the daemon side to avoid the bun install hardlink race that affects parallel installs against a shared cache.

Reproduce locally

# Rehearse vs act fixtures
$ git clone https://github.com/plsft/rehearse && cd rehearse
$ pnpm install
$ pnpm tsx bench/compare.ts --skip-cold

# Rehearse vs GitHub Actions on plsft/typey (9-cell matrix)
$ git clone https://github.com/plsft/typey && cd typey
$ time rehearse run .github/workflows/ci.yml
$ time rehearse run --remote .github/workflows/ci.yml
$ git push  # GH Actions runs in the background, see Actions tab

Honest caveats

  • Numbers vary by region, upstream-registry latency, and time of day. Treat the ratios as load-bearing, not absolute milliseconds.
  • The hono-node-matrix target currently regresses on GH-hosted runners (works locally; under investigation). Excluded from the headline.
  • Bench was last refreshed for v0.4.0 against the v0.3.11 baseline. Per-OS macOS/Windows numbers (host backend only — no Docker on those GH runners) in bench/RESULTS.md.
  • Rehearse's local run uses host bun + warm node_modules. If you wipe both, the cold run is closer to act in absolute time (~10-15s on the typey demo) — though still meaningfully faster, and the warm-loop gap is what matters for daily iteration.

Full methodology + per-OS table: bench/RESULTS.md on GitHub.
