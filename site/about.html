Free OSS core. Optional hosted offload.

Rehearse is two things: an Apache 2.0 local-first runner for GitHub Actions that runs on your laptop with no signup, no telemetry, and no network dependency — and an optional paid hosted layer (Rehearse Pro) for when the laptop isn't enough. Same YAML targets both; you opt into Pro with a single --remote flag.

What Rehearse is

Rehearse started as a question: your CI loop is `git push → wait → fail → repeat`. What if it ran on every save instead? That answer turned into a runner that reads your .github/workflows/*.yml, executes it locally, and finishes 5–30× faster than act on standard workflows — 30× on services, where act doesn't actually complete. Same YAML, no new config.

The OSS core is two npm packages today: @rehearse/cli (the rh binary that runs your workflows + an rh ci subcommand tree for TypeScript-authored pipelines) and @rehearse/ci (the typed builders that compile to standard GitHub Actions YAML). Two earlier packages — @rehearse/runner and @rehearse/git-core — were folded into @rehearse/cli in v0.6.0; their last published versions stay on npm forever for back-compat with old pre-push hooks. All Apache 2.0.

Rehearse Pro is the optional paid layer: a single-tenant microVM (Fly Machines via the Sprites runtime) you offload to when your laptop can't keep up with a multi-cell matrix. Same workflow YAML; switch with rh run --remote. Pricing is in $49 / 25,000-CPU-minute blocks — predictable, no per-minute meter. The OSS runner stays free forever and Pro is never required; rh run on a laptop is the primary use case.

What Rehearse isn't

  • Not a fork or rewrite of act. Different architecture (host backend in addition to container; per-cell git worktrees for matrix; in-process action shims), different goals (pre-push and watch are first-class). The bench documents the deltas.
  • Not a hosted-runner replacement like Blacksmith / Ubicloud / RunsOn. Those are post-push services that swap GitHub-hosted runners for faster ones — orthogonal to what Rehearse does. Pro is similar in shape but the OSS runner is what defines the product, and it runs offline.
  • Not telemetry-instrumented. The OSS runner doesn't phone home — compat audits, bench numbers, and provenance stay on your machine. Pro's API logs run metadata to gate billing + capacity, audited at /security.
  • Not locked to Pro. Drop the --remote flag and the same workflow runs locally. Your CI keeps working if Pro disappears, costs change, or you stop paying.
  • Not a wrapper around the git binary's protocol bits. The pure-TypeScript git protocol implementation is folded into @rehearse/cli as of v0.6.0; the standalone @rehearse/git-core package is frozen at v0.5.4 for users who want the protocol pieces alone.
