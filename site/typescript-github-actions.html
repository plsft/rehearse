// Why YAML for CI breaks down

YAML works until it doesn't.

A small workflow as YAML is fine. Past ~150 lines, the cracks show. Past ~500 lines, you're maintaining a config language with none of the affordances of a real one:

  • 1
    No types, no autocomplete. Did you misspell runs-on as runs_on? You find out 90 seconds into a run on push. Did you pass node-version: 20.0.0 when setup-node wants '20'? Same. The CI itself is where you discover your config is wrong.
  • 2
    No refactor. Renaming a job key cascades: needs: references, if: needs.X.outputs.Y expressions, ${{ matrix.X }} substitutions, downstream deploy gates. Find/replace works until it silently breaks something that compiles fine but means the wrong thing.
  • 3
    No reuse. You can extract composite actions and reusable workflows, but the YAML cost of doing so is high (separate file, separate schema, plumbing inputs by name). Most teams just copy-paste.
  • 4
    Expression escaping is medieval. ${{ }} inside ${{ }}, GitHub-flavored JSON inside YAML strings, double-escaped backslashes in regex inside heredocs. TypeScript template literals compose cleanly; YAML strings do not.
  • 5
    No code-review affordance. Diffing a YAML refactor is hostile. Reviewers can't tell which changes are semantic vs cosmetic. TypeScript diffs at least benefit from typed builders making intent legible.
// Forward path · TS → YAML

100% lossless. Snapshot-tested.

Every public SDK feature compiles to canonical GitHub Actions YAML. We assert it via a snapshot test suite covering 6 reference workflows that exercise every construct GitHub Actions supports: 

ts-ci/test/compiler/snapshots/
├── basic-ci.yml             — pipeline + 2 jobs + needs
├── matrix-build.yml         — strategy.matrix with include + exclude
├── agent-aware.yml          — conditional steps with custom expr
├── monorepo.yml             — paths filter + outputs + dependencies
├── deploy-cloudflare.yml    — environment + secrets + permissions
└── full-featured.yml        — schedule + concurrency + permissions + cancel

Every time we ship a feature in @rehearse/ci, a snapshot is committed and assertions run on every CI commit. If your TS pipeline compiles, the resulting YAML is canonical — the same shape GitHub itself emits, verified against the actions/runner reference behavior.

What "lossless" means

Every construct GitHub Actions accepts is expressible in the SDK with a 1:1 compile output. None of these features are silently dropped or rewritten:

  • ✓ All trigger types (push, pr, workflow_dispatch with inputs, schedule, release, workflow_run)
  • ✓ Workflow + job + step env
  • ✓ Workflow + job permissions
  • ✓ Workflow + job-level concurrency
  • ✓ Workflow defaults (run.shell + working-directory)
  • ✓ Matrix (variables, include, exclude, fail-fast, max-parallel)
  • ✓ Services (image, env, ports, options)
  • ✓ Job needs, conditions, timeout-minutes, outputs
  • ✓ Job environment block (name + url)
  • ✓ Steps (run, action, checkout) with id, condition, env, continueOnError, with
  • ✓ Local + remote composite actions
  • ✓ Local reusable workflows
  • ✓ Camel→kebab auto-translation (idToken → id-token etc.)
  • ✓ Expression helpers (secrets(), github(), hashFiles(), etc.)
// Reverse path · YAML → TS migration

Adopt typed CI on an existing repo. One command.

npx rh ci convert .github/workflows/ci.yml --out .rehearse/pipelines/
# → .rehearse/pipelines/ci.ts

rh ci convert is a migration starter, not a faithful round-trip. Common shapes survive untouched — triggers, jobs, runner, steps with run/uses/with/env/if, job-level conditions and outputs, workflow permissions and env. Advanced constructs get hand-ported after the initial generation.

SURVIVES THE ROUND-TRIP
  • ✓ Triggers (push / pr / schedule / workflow_dispatch — without inputs)
  • ✓ Jobs with name, needs, condition, env, timeout-minutes, outputs
  • ✓ Runner — Runner.github('ubuntu-latest') recognized; matrix expressions fall back to Runner.custom(...)
  • ✓ Steps (checkout / action with with / run with id, condition, env, continueOnError)
  • ✓ Workflow-level permissions + env
  • ✓ name: on every step preserved
DROPPED — HAND-PORT AFTER CONVERT
  • ○ Matrix block (entire strategy:)
  • ○ Services (postgres, redis, etc.)
  • ○ Concurrency
  • ○ Defaults (run.shell, run.working-directory)
  • ○ Job-level permissions (when different from workflow-level)
  • ○ Environment block (name + url object form)
  • ○ workflow_dispatch inputs (type/options/required)

Why honest about the gap: the SDK supports authoring everything in the dropped column — only the YAML→TS converter doesn't auto-extract them yet. Common adoption flow: convert once, hand-port the dropped blocks (usually 5–15 min), then live in the TS source-of-truth forever. rh ci compile round-trips the hand-ported version back to YAML to verify.

// What the SDK looks like

Real workflow. ~50 lines of typed TS.

A 9-cell matrix CI workflow with prettier + eslint + typecheck + test + build. This compiles to a 50-line YAML file that runs on stock GitHub Actions: 

import { job, pipeline, Runner, step, triggers } from '@rehearse/ci';

export const ci = pipeline('CI', {
  triggers: [triggers.pullRequest(), triggers.push({ branches: ['main'] })],
  jobs: [
    job('test', {
      runner: Runner.github('${{ matrix.os }}'),
      matrix: {
        variables: {
          os: ['ubuntu-latest', 'macos-latest', 'windows-latest'],
          'bun-version': ['1.2.x', '1.3.x', 'latest'],
        },
        failFast: false,
      },
      steps: [
        step.checkout(),
        step.action('oven-sh/setup-bun@v2', {
          name: 'Setup Bun ${{ matrix.bun-version }}',
          with: { 'bun-version': '${{ matrix.bun-version }}' },
        }),
        step.run('bun install --frozen-lockfile', { name: 'Install' }),
        step.run('bunx prettier --check .',    { name: 'Format' }),
        step.run('bunx eslint .',              { name: 'Lint' }),
        step.run('bunx tsc --noEmit',          { name: 'Typecheck' }),
        step.run('bun test',                  { name: 'Test' }),
        step.run('bun build src/index.ts --outdir dist --target bun', { name: 'Build' }),
      ],
    }),
  ],
});

Every identifier is typed. Runner.github('not-a-real-runner') compiles, but step.action's with: is typed against the action's known input schema where we have it (the language presets). Matrix variable names autocomplete in ${{ matrix.X }} contexts in IDEs that support TypeScript template-literal type narrowing.

// vs adjacent tools

vs Pulumi / Earthly / Dagger.

Other "CI in code" tools exist. Where @rehearse/ci sits relative to them:

Tool Output Lock-in?
@rehearse/ci Standard GitHub Actions YAML None — compiled YAML runs without us
Pulumi (with @pulumi/github) GH Actions YAML, but as part of a larger IaC stack Pulumi runtime + state backend required for full IaC features
Earthly Earthfiles (their own DSL, not GH Actions YAML) Yes — your CI runs on Earthly's runtime
Dagger Dagger pipeline programs (TS / Go / Python), not GH Actions YAML Yes — your CI runs on Dagger's runtime (locally or in cloud)
Hand-rolled YAML generators (yq, jinja, etc.) Whatever you template None — but no types, no autocomplete, no IDE help

@rehearse/ci's niche: typed TypeScript that compiles lossless to standard GitHub Actions YAML, with zero runtime dependency on us. If Earthly or Dagger fits your runtime story, use those — they give you more abstraction in exchange for living in their execution model.
