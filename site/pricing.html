// faq

Common questions.

What about my source code and secrets?

Your code is shipped to the VM at run time and lives in your dedicated 100 GB persistent volume. Each customer gets a hardware-isolated microVM — no shared kernel, no cross-tenant access. Secrets are passed in via env at run time and aren't logged.

Can I cancel anytime?

Yes. Stripe Customer Portal handles cancellation, refund (14-day window), card update, plan switch. We never gate it.

What's the 25,000-minute cap and how does it work?

The cap is on active CPU minutes — the actual time the VM is executing your workflow. Idle time is free; you only burn minutes when runner run --remote is doing real work on your VM. Email alerts fire at 75% and 90% of your cap. Hit 100% and your runs pause until the cap resets (first of next UTC month) OR you add another $49 / 25k block. To add a block, just reply to the alert email — I'll bump your cap within an hour.

What if I outgrow 25,000 minutes?

Each additional $49 / 25,000-minute block stacks on your base. Two blocks = $98/mo for 50k minutes; three = $147/mo for 75k. No tier ladders, no seat surcharges. If you're consistently above 100k min/mo, talk to George about Enterprise — bulk pricing makes more sense at that volume.

How long does provisioning take?

After Stripe checkout the welcome email arrives within ~30 seconds, but your VM is still spinning up — the actual provisioning runs in a GitHub Actions workflow that takes 60–120 seconds. The first runner run --remote waits briefly if the VM isn't ready yet; subsequent runs are warm.

Will the OSS runner ever be paywalled?

No. Apache 2.0, free, supported. Pro is strictly additive — pass --remote with REHEARSE_TOKEN set and the runner ships your workflow to your VM; drop the flag and you're back to running locally.

Do you have a free trial?

Implicitly yes — 14-day refund, no questions, the Stripe Customer Portal handles it. Sign up, evaluate honestly, refund if it's not worth it. We'd rather have you leave than feel stuck.

Is the OSS runner the same binary as Pro?

Yes. @rehearse/runner on npm is one Apache 2.0 binary. Run it locally for free; pass --remote with a Pro token and the same binary ships your workflow to your VM. Nothing about Pro changes the OSS runner — Pro is a hosted execution target, not a fork.

What languages does the Pro VM support?

All 11 verified end-to-end against a real Pro account: Node.js, Bun, Deno, Python, Go, Java, Ruby, Rust, Elixir, Erlang, and .NET (8 / 9 / 10). The first 10 are preinstalled on the VM; .NET 8, 9, and 10 install side-by-side via Microsoft's dotnet-install.sh on first actions/setup-dotnet step (~16s per channel first time, ~7ms cached thereafter — multi-channel matrix supported). PHP is the one current gap — no managed install path on the VM today; PHP workflows still work great on the OSS runner locally. Workflows that declare services: (Postgres, Redis) need Docker on the VM, which isn't on the default tier; use the OSS container backend locally for those.
