// faq

Common questions.

What about my source code and secrets?

Your code is shipped to the sprite at run time and lives in your dedicated 100 GB persistent volume. Each customer gets a hardware-isolated microVM — no shared kernel, no cross-tenant access. Secrets are passed in via env at run time and aren't logged.

Can I cancel anytime?

Yes. Stripe Customer Portal handles cancellation, refund (14-day window), card update, plan switch. We never gate it.

Are there usage limits?

Not at v0 launch. Your sprite is provisioned with 100 GB persistent storage at the Sprites.dev default compute tier; that's the natural ceiling. If you're running CI heavily and need more, talk to me before I have to email you about it. We'll figure it out before adding any cap.

How long does provisioning take?

After Stripe checkout the welcome email arrives within ~30 seconds, but your sprite is still spinning up — the actual provisioning runs in a GitHub Actions workflow that takes 60–120 seconds. The first runner run --remote waits briefly if the sprite isn't ready yet; subsequent runs are warm.

Will the OSS runner ever be paywalled?

No. Apache 2.0, free, supported. Pro is strictly additive — pass --remote with REHEARSE_TOKEN set and the runner ships your workflow to your sprite; drop the flag and you're back to running locally.

Do you have a free trial?

Implicitly yes — 14-day refund, no questions, the Stripe Customer Portal handles it. Sign up, evaluate honestly, refund if it's not worth it. We'd rather have you leave than feel stuck.

Is the OSS runner the same binary as Pro?

Yes. @rehearse/runner on npm is one Apache 2.0 binary. Run it locally for free; pass --remote with a Pro token and the same binary ships your workflow to your sprite. Nothing about Pro changes the OSS runner — Pro is a hosted execution target, not a fork.

What languages does the Pro sprite support?

All 11 verified end-to-end against a real Pro account: Node.js, Bun, Deno, Python, Go, Java, Ruby, Rust, Elixir, Erlang, and .NET. The first 10 are managed natively by Sprites.dev (preinstalled toolchains); .NET is installed via Microsoft's dotnet-install.sh on first actions/setup-dotnet step (~16s first time, ~7ms cached thereafter). PHP is the one current gap — Sprites.dev doesn't ship PHP and there's no clean install path; PHP workflows still work great on the OSS runner locally. Workflows that declare services: (Postgres, Redis) need Docker on the sprite, which isn't on the default tier; use the OSS container backend locally for those.
