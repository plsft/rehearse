★ LEAD PACKAGE — UNIQUE IN MARKET

@rehearse/ci

library · npm · source

Author GitHub Actions pipelines in TypeScript with type safety and IDE autocomplete, then compile to plain GitHub Actions YAML. The generated YAML has zero runtime dependency on us — you can eject any time. No real competitor exists for this — hosted-runner SaaS (Blacksmith, Ubicloud, RunsOn) doesn't try; YAML alternatives lock you into a proprietary execution model.

Why it exists

YAML for CI is unmaintainable past a few hundred lines. No types, no autocomplete, copy-paste errors that fail in CI 90 seconds later. This SDK gives you typed builders that compile to standard GitHub Actions YAML — the forward path is 100% lossless and snapshot-tested.

What it does

  • ✓ Typed builders: pipeline(), job(), step.run() / .checkout() / .action(), triggers.push() / .pullRequest() / .schedule(), Runner.github() / .selfHosted() / .custom()
  • ✓ Context helpers: secrets(), vars(), github(), env(), needs(), steps(), expr(), hashFiles() — all validated, throw on empty/whitespace
  • ✓ Language presets: node, bun, python, rust, go, docker — each exporting setup() / install() / test() / build() / lint() with correct action references
  • ✓ Hand-rolled YAML serializer — no external YAML library at runtime
  • ✓ YAML → TypeScript migration starter (convert()) — handles common shapes (push/pr/dispatch/schedule, run/uses/with/env/if). Drops matrix / services / concurrency / job-level permissions; emits warnings. Use it to bootstrap, then hand-port the advanced bits.
  • ✓ Cost estimator (estimate()) — per-run and monthly GitHub-hosted CI cost from a list-price snapshot baked into the package. Math is real (per-job × per-min × runs/month against the public price list); price tables refresh per release.

How it's used

// .rehearse/pipelines/ci.ts
import { pipeline, job, step, triggers, Runner } from '@rehearse/ci';
import { node } from '@rehearse/ci/presets';

export const ci = pipeline('CI', {
  triggers: [triggers.pullRequest(), triggers.push({ branches: ['main'] })],
  jobs: [
    job('test', {
      runner: Runner.github('ubuntu-latest'),
      steps: [step.checkout(), node.setup('20'), node.install(), node.test()],
    }),
  ],
});

Compile via rh ci compile (uses @rehearse/cli) → produces .github/workflows/ci.yml.

Numbers

  • Snapshot tests covering every preset and the YAML compiler — every output is committed
  • Zero runtime dependencies. The yaml package is a dev-only dep used by the converter (YAML → TS direction)
PACKAGE

@rehearse/runner

binary: runner · npm · source

The headline product. Reads your existing .github/workflows/*.yml and executes them on your laptop in seconds — same YAML, no edits. 5–9× faster than act on standard workflows, 30× on services where act doesn't finish.

Why it exists

git push → wait 90 seconds → CI fails → fix → push → wait → fail. That's the slowest inner loop developers have. The runner reads the same YAML and runs it locally so the loop closes in seconds.

What it does

  • ✓ Two backends per job: host (subprocess, fast, reuses your node_modules) and container (Docker, parity with GH-hosted)
  • ✓ Matrix in parallel via per-cell git worktree — three Node versions in three isolated checkouts, no shared-workspace races
  • ✓ services: on a private Docker network with --network-alias; health-check waiting included
  • ✓ actions/cache semantics on local fs — exact-key + restore-key longest-prefix; cache-hit / cache-primary-key / cache-matched-key outputs
  • ✓ actions/upload-artifact / download-artifact backed by .runner/artifacts/
  • ✓ Composite actions — local ./.github/actions/* and remote org/repo[/sub]@ref, auto-cloned
  • ✓ Local reusable workflows (uses: ./.github/workflows/foo.yml at job level) with with: + secrets: inherit
  • ✓ JS actions (runs.using: node20|18|16) on the host backend — auto-cloned at the requested ref, full INPUT_* / GITHUB_OUTPUT contract. Container-backend execution of JS actions is on the roadmap.
  • ✓ 16 in-process shims: checkout, setup-{node,python,go,java,dotnet,bun}, setup-pnpm, setup-deno, rust-toolchain, setup-ruby, cache, upload/download-artifact, codecov, github-script
  • ✓ if: on jobs and steps with ${{ matrix.* }}, env, secrets, needs, steps, runner, github, inputs
  • ✓ needs: with topological scheduling and bounded parallelism

Four CLI subcommands

runner run <workflow.yml>     # execute a workflow
runner watch <workflow.yml>   # re-run on every save (debounced, queue-coalesced)
runner install-hook           # write .git/hooks/pre-push that runs your workflow
runner compat <workflow.yml>  # static audit: how much would actually execute

Programmatic API

import { run, compat } from '@rehearse/runner';

const result = await run({
  workflowPath: '.github/workflows/ci.yml',
  jobFilter:    'test',
  backend:      'auto',
  maxParallel:  4,
});

const audit = compat('.github/workflows/ci.yml');
// → { coverage, jobs, stepsTotal }

Numbers

  • 121 tests across 13 suites, all green
  • Compat: 100% on our own CI · 96.15% on vitejs/vite · 97.18% on honojs/hono
  • vs act (Linux GH-runner v0.4.0 warm — see bench/RESULTS.md): 5.23× on our-ci, 8.99× on node-matrix, 32.82× on service-postgres
  • Runtime deps: commander, picocolors, yaml (plus workspace siblings)

Honest limits

  • ○ Remote reusable workflows (org/repo/.github/workflows/foo.yml@ref) — local form works
  • ○ Docker actions (runs.using: docker) — JS-action runtime ships
  • ○ OIDC / id-token: write
  • ○ concurrency: cancellation
PACKAGE — THE MIGRATION ON-RAMP

@rehearse/cli

binary: rh · npm · source

The rh command — companion CLI for the TypeScript-pipeline workflow. The killer feature: rh ci convert turns your existing GitHub Actions YAML into typed TypeScript in one command. Adopt typed CI on a real repo without rewriting a thing.

Six subcommands

rh ci init      # scaffold .rehearse/pipelines/ci.ts + .rehearse/package.json + rehearse.config.mjs
rh ci convert   # ★ existing .github/workflows/foo.yml → .rehearse/pipelines/foo.ts (the migration on-ramp)
rh ci compile   # .rehearse/pipelines/**/*.ts → .github/workflows/*.yml
rh ci validate  # type-check pipelines without writing output
rh ci watch     # recompile on save (inner loop for pipeline authoring)
rh ci estimate  # GitHub-hosted CI cost per run and per month, given step durations

Why it's separate from @rehearse/ci

@rehearse/ci is a library — you import from it and write TypeScript. The CLI wraps that library in commands that handle the file-system side: walking .rehearse/pipelines/, importing .ts files via tsx, writing .github/workflows/*.yml, parsing existing YAML for the reverse direction. Separating these means @rehearse/ci stays runtime-dep-free — tsx would otherwise pull in chokidar, esbuild, and others.

Stack detection

rh ci init detects your project (bun.lockb → Bun, Cargo.toml → Rust, go.mod → Go, etc.) and scaffolds an appropriate template. Falls back to a generic template for unknown stacks.

Module-resolution decoupling

rh ci init writes a .rehearse/package.json with {"type": "module"} to scope ESM resolution to the pipelines subtree — your host project's package.json can be CJS, ESM, or unset and rh ci compile still works. The config file is rehearse.config.mjs (extension-explicit ESM) for the same reason.

PACKAGE — SPECIALTY (SANDBOXED RUNTIMES)

@rehearse/git-core

library · npm · source

Full git protocol implementation in TypeScript with zero native dependencies. The unique-fit use case: anywhere the system git binary isn't an option — Cloudflare Workers, browsers, Deno Deploy, edge runtimes, embedded TS environments. Ships standalone — install it directly when you need git inside a sandboxed JS runtime.

Why it exists

Most git tooling shells out to the git binary. That doesn't work inside Cloudflare Workers, the browser, or any sandboxed runtime. It's also slow (process spawn per operation) and inflexible (you get whatever the user's git version supports). @rehearse/git-core is a complete from-scratch reimplementation in TypeScript.

Relationship to the runner

The runner's actions/checkout shim is a no-op for local execution because your repo is already in cwd — cloning it again would be wasted I/O. The Pro VM clones via the standard git binary on the VM today. We ship git-core as the foundation it'll use when a real in-process clone is needed (e.g., remote checkout from a Worker, or with: repository: pointing at a different repo than cwd).

What it does

  • ✓ Git objects: read/write blobs, trees, commits, tags. Loose-object format and SHA-1 hashing. zlib via pako
  • ✓ Refs: refs/heads/, refs/tags/, refs/remotes/, packed-refs file format — read, write, list, dereference
  • ✓ Packfiles: parse v2 packfiles (REF_DELTA + OFS_DELTA), write packfiles, packfile index v2 (.idx)
  • ✓ Diff & merge: Myers diff, three-way merge, patch generation, hunk-based representation, conflict markers
  • ✓ Smart HTTP protocol: clone over git-upload-pack and git-receive-pack v2 — capability advertisement, want/have negotiation, sideband-64k chunking
  • ✓ HTTP client via plain fetch() — Workers, browsers, Node, Bun

Programmatic API

import { GitClient } from '@rehearse/git-core';

const client = new GitClient({ fetch });
const result = await client.clone({
  url:   'https://github.com/honojs/hono.git',
  ref:   'main',
  depth: 1,
});
// → { objects, refs, head }

// or compose lower-level primitives
import { parseTree, parsePackfile, computeDiff } from '@rehearse/git-core';

Numbers

  • 162 tests across 8 suites — refs, merge, diff, protocol, objects, packfile, http-flow, client-integration
  • One runtime dep: pako (zlib in JS)
  • No native deps, no shellout, no git binary required

Where it's used

  • Anywhere you need git inside a JS-only sandbox: edge workers, browser-based code editors, embedded VCS UIs
  • Future runner work: in-process checkout for remote refs, Worker-based runners
// How they fit together

Two pipelines, one engine.

The runner reads YAML and executes it. The rh CLI compiles TypeScript pipelines down to YAML — and that YAML feeds back to the runner, or to GitHub directly, or both. git-core ships separately as a standalone pure-TS git library for sandboxed runtimes.
