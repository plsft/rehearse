Pre-push CI: how to run GitHub Actions before git push

TL;DR

Stop the git push → wait → fail → repeat loop. Two ways to run your GitHub Actions workflow locally before pushing: (1) rehearse watch ci.yml reruns on every file save (sub-second on warm cache); (2) rehearse install-hook writes a .git/hooks/pre-push that gates the push on a green local run. Same YAML, no edits. Free.

The problem with post-push CI

The default GitHub Actions loop is broken at the developer level:

  1. You finish a change.
  2. git commit && git push.
  3. Wait 30–120 seconds for GitHub Actions to spin up a runner.
  4. Wait another 30–60 seconds for the workflow to install deps + run tests.
  5. See a red ✗ on a typo or missing import.
  6. Fix it locally. git commit --amend && git push -f.
  7. Repeat.

Each cycle costs 60–180 seconds of wall time and burns through your branch's commit history with `wip: try fix` noise. With a typical fix needing 2–4 cycles, you've spent 5–15 minutes waiting for feedback you could have had in ~1 second if CI ran on your laptop.

Option 1: rehearse watch — CI on every save

The most aggressive (and most useful) option: have your editor's save trigger a CI rerun. Install @rehearse/runner and run: 

npm install -g @rehearse/runner
cd ~/your-repo
rehearse watch .github/workflows/ci.yml

Now any file save in the repo reruns the workflow. The runner debounces rapid saves and coalesces queued runs (a save while a run is mid-flight queues exactly one next-run, not N) so you don't burn cycles on burst-typing. On warm cache (deps already installed, build artifacts present), reruns finish in well under a second for trivial workflows and 1–5s for typical CI.

Why this matters: the cycle from "save" to "see CI pass or fail" closes inside your IDE workflow. You don't context-switch to a terminal. You don't context-switch to a browser tab. You don't burn GitHub Actions minutes. The signal is right there.

Option 2: rehearse install-hook — pre-push gate

If you're not ready for "CI on every save" but want a safety net, install the pre-push git hook: 

cd ~/your-repo
rehearse install-hook
✓ wrote .git/hooks/pre-push (will run .github/workflows/ci.yml before push)

Now git push runs your CI workflow first. If CI is green, the push proceeds. If red, the push is blocked and you see the failure immediately — no GitHub Actions minutes burned, no broken commit landing on your branch, no notification to your team about a failed PR check.

Bypass the hook (rare cases like an emergency deploy) with git push --no-verify. Disable it permanently by deleting .git/hooks/pre-push.

Combining both: max-coverage workflow

The ideal setup uses both:

  • rehearse watch running in a terminal pane while you code → save-triggered, sub-second feedback
  • rehearse install-hook as a safety net → catches anything you forgot before push reaches GitHub

Net effect: most CI failures never reach GitHub Actions. You get the iteration speed of unit-test-style local feedback with the coverage of full CI.

Why hosted-runner services can't do this

Hosted-runner SaaS services like Blacksmith, Ubicloud, RunsOn, Depot, WarpBuild all start the clock at git push. They're faster GH-hosted runners — same loop position, just less expensive per minute.

They structurally cannot do watch-on-save: they bill per minute, every save would be a new run, and the bill would be unworkable. Pre-push CI requires execution to happen on hardware you already own — which is what a local runner does.

FAQ

Does this work with my existing .github/workflows/*.yml?

Yes. Rehearse reads the same YAML GitHub Actions reads — same matrix, services, composite actions, JS actions, cache, conditional steps. ~97% of real workflows just work without edits.

What if my workflow needs services: postgres?

Works locally with the container backend (Docker required for that job). Health-check waiting, network alias, all wired correctly.

What about secrets?

Pass via environment, or via --env-file .env. Rehearse expands them as ${{ secrets.X }} the same way GitHub Actions does.

How is this different from running npm test locally?

Your CI workflow does more than npm test — it runs lint, typecheck, build, integration tests, possibly across a matrix of Node versions. Re-implementing all of that in your terminal is a maintenance burden. Rehearse just runs the same YAML you already have.

Try it: npm install -g @rehearse/runner && rehearse install-hook. 30 seconds to a green-CI-or-blocked-push setup.
