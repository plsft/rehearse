TypeScript GitHub Actions: from YAML to typed pipelines

TL;DR

Author your GitHub Actions workflows in typed TypeScript with full IDE autocomplete and refactor support. Compile to standard GitHub Actions YAML you commit alongside the TS source. @rehearse/ci's forward path is 100% lossless (snapshot-tested), and the migration on-ramp from existing YAML is one command: rh ci convert. The compiled YAML works on stock GitHub Actions with zero runtime dependency on the SDK. Apache 2.0.

YAML for CI breaks down at scale

A 50-line CI workflow in YAML is fine. A 500-line one is a nightmare. A 1500-line monorepo CI with matrix expansion, conditional jobs, reusable workflows, composite actions, and complex secret-passing is where teams burn weeks of engineering time on what should be config maintenance.

The specific failure modes:

  1. No types, no autocomplete. Misspell runs-on as runs_on? Find out 90 seconds into a run on push. Pass node-version: 20.0.0 when setup-node wants '20'? Same.
  2. No refactor. Renaming a job key cascades through needs: references, if: needs.X.outputs.Y expressions, ${{ matrix.X }} substitutions. Find/replace works until it silently breaks.
  3. Expression escaping is medieval. ${{ }} inside ${{ }}, GitHub-flavored JSON inside YAML strings, double-escaped backslashes inside heredocs.
  4. No reuse without ceremony. Composite actions and reusable workflows exist but the YAML cost of extracting them is high.
  5. Hostile diffs. Reviewing a YAML refactor PR is brutal — semantic vs cosmetic changes bleed together.

The TypeScript escape hatch

@rehearse/ci gives you typed builders for GitHub Actions: 

// .rehearse/pipelines/ci.ts
import { pipeline, job, step, triggers, Runner } from '@rehearse/ci';
import { node } from '@rehearse/ci/presets';

export const ci = pipeline('CI', {
  triggers: [triggers.pullRequest(), triggers.push({ branches: ['main'] })],
  jobs: [
    job('test', {
      runner: Runner.github('ubuntu-latest'),
      matrix: { variables: { node: ['18', '20', '22'] } },
      steps: [
        step.checkout(),
        node.setup('${{ matrix.node }}'),
        node.install(),
        node.test(),
      ],
    }),
  ],
});

Compile with rh ci compile (from @rehearse/cli). Output: standard .github/workflows/ci.yml committed to your repo. The compiled YAML is canonical — same shape GitHub itself emits — and runs on stock GitHub Actions with zero runtime dependency on Rehearse.

100% lossless forward path

Every public SDK feature compiles to canonical GitHub Actions YAML. We assert this via a snapshot test suite covering 6 reference workflows that exercise every construct GitHub Actions supports — matrix (variables, include, exclude, fail-fast, max-parallel), services (image, env, ports, options), concurrency, defaults, environment blocks, workflow + job + step env, workflow + job permissions, conditions, outputs, schedule triggers, workflow_dispatch with inputs, all standard step constructs, composite actions (local + remote), local reusable workflows, and camel→kebab auto-translation (idToken → id-token etc.).

Forward path losslessness means: anything you can do in YAML, you can do in TS, and the compiled YAML is exactly what GitHub expects. No silent translations, no semantic drift.

Migration on-ramp: one command

Adopting typed CI on a repo that already has .github/workflows/*.yml? 

npm install -D @rehearse/ci @rehearse/cli
npx rh ci convert .github/workflows/ci.yml --out .rehearse/pipelines/
# → .rehearse/pipelines/ci.ts

rh ci convert is a migration starter, not a faithful round-trip. Common shapes survive — triggers, jobs with name/needs/condition/env/timeoutMinutes/outputs, runner, all standard step constructs, workflow-level permissions and env. Advanced constructs that get hand-ported (matrix, services, concurrency, defaults, job-level permissions when different from workflow-level, environment block in object form, workflow_dispatch inputs) take 5–15 minutes per workflow.

The SDK supports authoring all of those constructs — only the converter doesn't auto-extract them yet. The common adoption flow: convert once, hand-port the dropped blocks, then live in the TS source-of-truth forever. rh ci compile round-trips back to YAML to verify.

Zero runtime dependency

The compiled .github/workflows/ci.yml has no runtime dependency on @rehearse/ci. npm uninstall @rehearse/ci and your CI keeps running on stock GitHub Actions exactly as before. The TS source is authoring-time only.

That's the key difference from frameworks like Pulumi, Earthly, or Dagger — those either compile to formats GitHub doesn't natively run, or require their own runtime to execute. @rehearse/ci compiles to YAML GitHub already runs.

FAQ

Can I use this without committing to Rehearse?

Yes. Compile your TS pipelines to YAML, commit both, then never run @rehearse/ci again if you don't want to. The compiled YAML works on stock GitHub Actions forever.

Does it support the language I use?

The presets cover Node, Bun, Python, Rust, Go, and Docker out of the box. Anything else uses generic step.run() and step.action() — no language-specific support needed.

How does this compare to writing CI in Pulumi or Earthly?

Pulumi compiles to GH Actions YAML but as part of a larger IaC stack with state-backend requirements. Earthly outputs Earthfiles (their own DSL, not standard GH Actions) and runs on Earthly's runtime. Dagger runs pipelines on its own runtime. @rehearse/ci's niche is typed TS that compiles lossless to standard GitHub Actions YAML with zero runtime dependency on us.

Is the SDK production-ready?

Yes. v0.6.18 ships with 40 unit tests + 6 snapshot tests covering every public construct. The forward path is the production code path; we use it for our own CI.

Try it: npm install -D @rehearse/ci @rehearse/cli && npx rh ci init. Scaffolds a typed pipeline based on your repo's stack.
