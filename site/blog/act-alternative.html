act alternative: a faster local GitHub Actions runner

TL;DR

If you've outgrown nektos/act — speed, services networking, watch mode, or "I don't want to install Docker just to run my CI" — Rehearse is the closest replacement. Same job (run .github/workflows/*.yml locally), 5–30× faster on the bench, a host backend that doesn't need Docker, real services: networking, a rehearse watch save-loop, and a pre-push git hook. Same YAML, no edits. Apache 2.0. Install: npm i -g @rehearse/runner.

Why look for an act alternative?

nektos/act is the OSS pioneer for running GitHub Actions workflows locally — ~56k GitHub stars and ~6 years of community use. It works for a lot of people. But there are real reasons teams look for an alternative:

  • It's slow on standard workflows. The published bench shows act at 63s on a 2-job typecheck-and-test pipeline; faster runners finish in 12s.
  • It's Docker-only. Every job spins a container, even when you'd be fine running on your host shell. That adds 5–15s of overhead per job and forces a Docker install.
  • services: blocks don't reliably finish. Workflows that declare a Postgres or Redis service routinely hang on health-check waits.
  • No watch mode. You re-run by hand every time you save.
  • No pre-push hook. If you want CI to gate git push, you write the script yourself.

What Rehearse does that act doesn't

Host backend (no Docker required)

Rehearse picks a backend per job. If your job doesn't declare services: and doesn't pin a container-only image, it runs steps as host subprocesses — using your already-installed Node, Bun, Python, etc. No image pull, no container spin-up, no second copy of node_modules. This alone is responsible for most of the speedup vs act.

Real services: networking

When your workflow does declare services: { postgres: ... }, Rehearse uses the container backend, starts the service on a private Docker network, wires --network-alias postgres so your job container can reach it as postgres:5432, and waits for the service's health check before running the first step. The published bench has a service-postgres workflow that act times out on at 360s; Rehearse runs it in ~11s.

Watch mode

rehearse watch .github/workflows/ci.yml reruns the workflow on every file save. Debounced and queue-coalesced — rapid edits collapse into a single rerun. Pair it with a CI workflow that includes lint + typecheck + test and your editor save becomes your CI trigger. Read more →

Pre-push git hook

rehearse install-hook writes a .git/hooks/pre-push that runs your CI workflow before letting git push complete. Push only if green. The CI minutes you save are real — broken pushes never reach GitHub Actions.

Migration: from act to Rehearse in 30 seconds

You don't need to change your YAML. Drop in: 

# Install
$ npm install -g @rehearse/runner

# Run any workflow that act ran
$ rehearse run .github/workflows/ci.yml

# Or filter to one job (same flag as act -j)
$ rehearse run .github/workflows/ci.yml --job test

# Or wire it into your inner loop
$ rehearse watch .github/workflows/ci.yml
$ rehearse install-hook

Where act is still the right call

Be honest about the alternative:

  • If you specifically need Docker actions (runs.using: docker) — act ships them, Rehearse doesn't yet.
  • If you're already invested in act's .actrc + per-image overrides and don't want to relearn anything.
  • If you want a single-binary Go tool instead of a Node CLI.

FAQ

Is Rehearse a fork of act?

No. Different architecture, different goals. Rehearse is written in TypeScript with a separate planner / scheduler / two-backend executor; act is a single Go binary that drives Docker. They share the category but no code.

Can I switch between act and Rehearse on the same repo?

Yes. They both read the same .github/workflows/*.yml with no edits. You can run the same workflow with both tools and compare.

Does Rehearse support strategy.matrix?

Yes — with parallel execution via per-cell git worktree. Each matrix cell gets an isolated checkout so cells never race on shared workspace state. include + exclude are honored.

What about actions/cache?

Real semantics on local fs. Exact-key + restore-key longest-prefix matching. cache-hit, cache-primary-key, cache-matched-key outputs. Persistent across runs.
